TradeFlow — How a Trade Decision Is Made

A walk-through of the existing decision flow, from a Binance WebSocket tick all the way to a paper order. Built by inspecting the code in src/.

The system is a 14-stage pipeline, but the stages are not in one file — they're spread across client React effects, server actions, and the AI orchestrator. This doc lays them out in execution order with the file you'll find each stage in, plus what it produces and what happens when it fails.
succeeded degraded / fallback aborts pipeline required stage optional stage

1 · Architecture at a glance

Client (browser, React)

useMarketStore · Binance WebSocket → candles + tickers
AiDecisionSubscriber · polls per symbol every ~96 s
AiExecutionEngine · risk-gates and fires orders
useMatchingEngine · monitors SL / TP / expiry
⇄

Server (Next.js server actions)

getStrategyDecision() · runs the strategy pipeline + LLM
submitDecisionJob() · AI orchestrator (priority queue + cache)
getSymbolSentiment() · Fear & Greed + news
createPaperOrder() / fillPaperOrder() / closePaperPosition() · DB-atomic

The browser tab is the heartbeat: nothing happens if no one has the page open. There is no server-side cron and no background worker.

LLM coordinator

serverrequired

The snapshot + indicators + sentiment + portfolio context (NOT raw candles) goes through the global AI orchestrator. It dedups by fingerprint, prefilters flat snapshots (synthetic HOLD = 0 tokens), tier-routes (premium 70 B for elite alignment ≥ 70, otherwise cheap 8 B), and runs Groq → OpenRouter → local fallback engine. Returns a MarketDecision with entry / SL / TP / setup-quality / confidence.

clientserver

Hard exits (SL / TP / expiry) → closePaperPosition with the right CloseReason. Soft exits (only for LLM-owned positions): setup graded Avoid, LLM flipped to opposite side at confidence ≥ 60, LLM upgraded to AVOID at ≥ 65, or persistent HOLD at ≥ 75 after > 30 min. 60 s per-symbol cooldown prevents thrash.

5 · The AI orchestrator (Stage 9 in depth)

Every LLM-bound call in the system funnels through one in-process scheduler. The orchestrator's job is to share one TPM budget across all callers, dedup identical requests, and never block the trading loop when an LLM is down.

Prefilter — skip the LLM entirely on flat snapshots

If alignment score < 30 AND |netDirection| < 8 AND no open position → return a synthetic HOLD without burning any tokens. Marked source: "prefilter".

Tier routing — premium vs cheap model

Alignment ≥ 70 → premium-tier model (e.g. 70 B). Otherwise the cheap-tier (8 B) handles the routine evaluation. Open-position re-evaluations also stay on cheap because they're fast SL/TP adjustments, not new theses.

Dedup + priority queue

Key = symbol:timeframe:regime:priceBucket:alignment. Concurrent submitters of the same key share one in-flight promise. The queue has 6 priority buckets (EXECUTION_CRITICAL → RECHECK) with FIFO inside each. SLAs prevent stale jobs from running (15-30 s).

Provider fallback chain

Configured primary (Groq or OpenRouter) → Groq cheap → OpenRouter primary → OpenRouter fallbacks (read from env) → local deterministic engine. The local engine reads the same snapshot and produces a defensively-sized decision, so trading keeps running in degraded mode.

Cache (TTL fingerprint)

Same fingerprint hits an in-memory cache: 90 s TTL for LLM answers, 20 s for fallback answers (so the chain retries sooner once provider cooldowns elapse). 64-entry LRU.

7 · Key data structures

StrategySnapshot — what the LLM reads

{
  symbol, timeframe, regime,
  netDirection:    -100…+100,   // signed weighted vote
  alignmentScore:  0…100,       // share of strategies agreeing
  topStrategies:   [ {strategyName, signal, confidence, weightedScore, reasoning[1]} × 5 ],
  conflictingStrategies: [ … × 3 ],
  aggregateMomentumScore, aggregateTrendScore, aggregateVolatilityScore,
  candlestickIntel?: { detections[], netBias, topConfidence, dominantCategory, narrative }
}

MarketDecision — what the LLM returns

{
  decision: "BUY" | "SELL" | "HOLD" | "AVOID" | "BREAKOUT LONG" | "BREAKDOWN SHORT" | "PULLBACK LONG" | "REVERSAL LONG",
  confidence:     0…100,
  setupQuality:  "A+" | "A" | "B+" | "B" | "C" | "Avoid",
  riskLevel:     "Low" | "Medium" | "High",
  executeTrade:   bool,
  positionSizePercent: 0…100,         // requested — the executor still caps
  expectedHoldTimeMinutes: 5…240,
  entryPrice, takeProfit, stopLoss,
  reasoning: string[1..4],
  warnings:  string[0..3],
  marketSummary: string
}

TradeAssessment — what the risk gate emits

{
  approved: bool,
  rejections: [{ code, message }],    // e.g. min_rr, low_confidence, book_full
  warnings:   [{ code, message }],
  metrics:    { expectedProfitPercent, expectedLossPercent, riskRewardRatio,
                entryDriftBps, volatilityScore },
  score:      { value: 0…100, grade: "A+"|"A"|"B"|"C"|"D", factors[] }
}

8 · The seams (where today's flow is fragile)

SeamWhat's there todayWhat's missing
One canonical pipeline runner Stages live in 3+ places (subscriber → server action → execution engine → matching engine) No single function you can call to run all 14 stages with explicit logging per stage
Server-side autonomy Browser must be open to decide or execute anything No scheduler / cron / background worker
Position management beyond hard SL/TP Soft-exit on AI flip is in evaluateExit(), client-side only No trailing stops, no breakeven-after-1R, no regime-flip auto-exit, no time-decay exit
Stage-14 analytics TradeHistory exists; win rate is computed in the UI No per-strategy / per-regime / per-AI-source win-rate tables. Decision attribution is buried in decisionMeta JSON
Per-stage audit trail Console logs only No DB row per pipeline execution; can't query "how often did sentiment fall back last week"
